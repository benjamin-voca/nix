# Machine Registry DSL — Design Document

**Date:** 2026-05-19  
**Status:** Design (not yet implemented)

---

## 1. Problem Statement

The current system has machines defined as NixOS module files in `modules/hosts/`, each calling `config.quad.lib.mkClusterHost { ... }` with explicit references to hardware modules, role modules, taints, and extra modules. Roles are defined as separate NixOS module files in `modules/roles/`. To add a new machine, you must:

1. Create a hardware module in `modules/hardware/`
2. Create a role module in `modules/roles/` (if new role)
3. Create a host definition in `modules/hosts/` that calls `mkClusterHost`
4. Wire it through `modules/imports.nix` auto-discovery

This works, but the **machine-level configuration is scattered across three directories** with no single source of truth for "what machines exist and what are their properties."

---

## 2. Proposed Directory Structure

```
machines/
├── default.nix              # Machine registry + role registry
├── hardware/                # Hardware modules (symlinked or moved from modules/hardware/)
│   ├── backbone-01.nix
│   └── frontline-01.nix
└── roles/                   # Role compositions (symlinked or moved from modules/roles/)
    ├── backbone.nix
    └── worker.nix           # Renamed from frontline.nix (clarifies intent)

# Existing, unchanged:
cluster/
  (remains in modules/outputs/bootstrap.nix — no changes to K8s resource generation)
```

**Key decision:** `cluster/` is NOT part of this DSL. K8s resources are cluster-wide, generated by `modules/outputs/bootstrap.nix`. The machine DSL is purely about NixOS host configuration.

### Symlink strategy (Phase 1)

During migration, `machines/hardware/` and `machines/roles/` can be symlinks back to `modules/hardware/` and `modules/roles/` respectively. This avoids moving files until Phase 3.

## 3. Nix Schema Design

### 3.1 `machines/default.nix`

```nix
# machines/default.nix
#
# Single source of truth for all machines in the cluster.
# Consumed by modules/outputs/nixos-configurations.nix to produce flake outputs.
{
  # ── Machine Registry ────────────────────────────────────────────────
  # Each entry maps 1:1 to a nixosConfiguration in the flake output.
  machines = {
    backbone-01 = {
      system = "x86_64-linux";
      hardware = ./hardware/backbone-01.nix;
      role = "backbone";         # key reference into roles block below
      sshHost = "backbone01";    # for deploy-rs (defaults to machine name)
      remoteBuild = true;        # cross-compile from aarch64-darwin

      # Taints applied to the K8s node (via kubelet --register-with-taints)
      taints = [
        { key = "role"; value = "backbone"; effect = "NoSchedule"; }
        { key = "infra"; value = "true"; effect = "NoSchedule"; }
      ];

      # Machine-specific modules that override/extend the role
      extraModules = [
        ({ lib, ... }: {
          boot.loader.grub.enable = lib.mkForce false;
          boot.loader.systemd-boot.enable = true;
          boot.loader.efi.canTouchEfiVariables = true;
          boot.loader.efi.efiSysMountPoint = "/boot";
        })
      ];
    };

    frontline-01 = {
      system = "x86_64-linux";
      hardware = ./hardware/frontline-01.nix;
      role = "worker";
      sshHost = "frontline01";
      remoteBuild = true;

      taints = [
        { key = "role"; value = "frontline"; effect = "NoSchedule"; }
      ];

      extraModules = [
        # frontline-01's cloudflared tunnel for SSH (machine-specific, not role-level)
        ({ pkgs, ... }: {
          systemd.services.cloudflared = { ... };  # as currently defined
        })
      ];
    };
  };

  # ── Role Registry ───────────────────────────────────────────────────
  # Roles are NixOS module files that compose profiles and services.
  # Keyed by role name, referenced by machines via `role = "backbone"`.
  roles = {
    backbone = {
      module = ./roles/backbone.nix;
      # Human-readable description (optional, for documentation)
      description = "Kubernetes control-plane node with ArgoCD, Harbor, Forgejo";
    };

    worker = {
      module = ./roles/worker.nix;
      description = "Kubernetes worker node with Docker";
    };
  };
}
```

### 3.2 Field Reference

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `system` | string | yes | — | Nix platform string (e.g. `"x86_64-linux"`) |
| `hardware` | path | yes | — | Path to hardware-configuration NixOS module |
| `role` | string | yes | — | Key into `roles` attrset |
| `sshHost` | string | no | machine name | Hostname for deploy-rs SSH |
| `remoteBuild` | bool | no | `false` | Whether deploy-rs should remote-build |
| `taints` | list | no | `[]` | K8s node taints |
| `extraModules` | list | no | `[]` | Additional NixOS modules (machine-specific overrides) |
| `secrets` | path | no | auto | SOPS secrets file (defaults to `secrets/${name}.yaml`) |

### 3.3 Role Field Reference

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `module` | path | yes | Path to the role's NixOS module |
| `description` | string | no | Human-readable description |

### 3.4 How This Maps to `mkClusterHost`

The registry is consumed by a new `modules/lib/machine-registry.nix` that replaces the per-host files. The function `mkClusterHost` itself remains **unchanged** — it's just called from a single place instead of scattered across `modules/hosts/*.nix`:

```nix
# modules/lib/machine-registry.nix (new file)
#
# Reads machines/default.nix and produces quad.hosts entries.
{config, inputs, ...}: let
  registry = import ../../machines/default.nix;
  inherit (registry) machines roles;

  # For each machine in the registry, call mkClusterHost with the right args.
  hostsFromRegistry = builtins.mapAttrs (name: machine: let
    role = roles.${machine.role};
  in
    config.quad.lib.mkClusterHost {
      inherit (machine) system;
      inherit name;
      hardwareModule = machine.hardware;
      roleModule = role.module;
      sshHost = machine.sshHost or name;
      remoteBuild = machine.remoteBuild or false;
      taints = machine.taints or [];
      extraModules = machine.extraModules or [];
    }
  ) machines;
in {
  config.quad.hosts = hostsFromRegistry;
}
```

### 3.5 How to Add a New Machine

**Before (current):**
1. Create `modules/hardware/newhost-01.nix`
2. Create `modules/hosts/newhost-01.nix` with full `mkClusterHost` call
3. File is auto-discovered by `modules/imports.nix`

**After (with registry):**
1. Create `machines/hardware/newhost-01.nix`
2. Add one entry to `machines/default.nix`:
   ```nix
   newhost-01 = {
     system = "x86_64-linux";
     hardware = ./hardware/newhost-01.nix;
     role = "worker";
   };
   ```

**Net reduction:** One fewer file to create, and the machine's properties are visible in a single file alongside all other machines.

### 3.6 How to Add a New Role

1. Create `machines/roles/newrole.nix` (a NixOS module importing profiles/services)
2. Add one entry to `machines/default.nix`:
   ```nix
   newrole = {
     module = ./roles/newrole.nix;
     description = "Description of the new role";
   };
   ```

## 4. Integration Points

### 4.1 `modules/imports.nix` Changes

Replace:
```nix
++ filesIn ./hosts
```
With:
```nix
++ [ ./lib/machine-registry.nix ]
```

The individual host files in `modules/hosts/` are no longer imported — the registry module generates all `quad.hosts` entries.

### 4.2 `modules/outputs/nixos-configurations.nix` — No Changes

This file already does `config.flake.nixosConfigurations = lib.mapAttrs (_: host: host) config.quad.hosts`. The registry module sets `config.quad.hosts` the same way the old host files did. Zero change needed.

### 4.3 `modules/outputs/deploy.nix` — No Changes

Already reads from `config.flake.nixosConfigurations` and `host._quad` for SSH/deploy config. The `mkClusterHost` function still attaches `_quad` metadata. Zero change needed.

### 4.4 `modules/outputs/checks.nix` — No Changes

The existing test infrastructure (`tests/nix/quad/hosts-output.test.nix`, VM tests) continue to work because they evaluate the same module system.

## 5. Golden Test Design

### 5.1 Recommended Approach: Option A — Compare Store Paths

**Why Option A:**
- NixOS `toplevel` derivations are content-addressed. If two evaluations produce the same store path, they are **bit-for-bit identical** — zero chance of silent semantic differences.
- Option B (JSON comparison) requires serializing the full config, which is fragile and produces huge diffs for irrelevant changes (ordering, etc.).
- Option C (eval comparison) is less precise — two different configs can have the same eval structure.

### 5.2 Test Strategy

The golden test validates that **the new registry produces identical NixOS configurations** to the current system. This is done by:

1. Building `nixosConfigurations.<name>.config.system.build.toplevel` from both the old and new code paths
2. Comparing the resulting store paths
3. Asserting they are equal

Since this is a migration test, it's temporary — once the old system is removed (Phase 3), the golden test is deleted.

### 5.3 Test Scaffold

```nix
# tests/nix/machines/golden-test.nix
#
# Validates that the machine registry DSL produces identical NixOS
# configurations to the current system.
#
# Run: nix-instantiate --eval --strict tests/nix/machines/golden-test.nix \
#        --arg pkgs 'import <nixpkgs> {}'
#
# Or via: nix flake check (once added to checks.nix)
{pkgs ? import <nixpkgs> {}}: let
  lib = pkgs.lib;

  # ── Old system: evaluate current modules ──────────────────────────────
  oldEval = lib.evalModules {
    specialArgs.inputs = {
      nixpkgs = pkgs;
      sops-nix = { nixosModules.sops = {}; };
      deploy-rs = { lib.x86_64-linux = { activate.nixos = x: x; }; };
      nixhelm = { packages.x86_64-linux = {}; chartsDerivations.x86_64-linux = {}; };
      nix-kube-generators = { lib = _: {}; };
      haumea = { lib = { load = x: {}; }; };
    };
    modules = [
      ../modules/options/flake.nix
      ../modules/options/quad.nix
      ../modules/lib/mk-cluster-host.nix
      ../modules/hosts/backbone-01.nix
      ../modules/hosts/frontline-01.nix
      ../modules/outputs/nixos-configurations.nix
    ];
  };

  # ── New system: evaluate via machine registry ─────────────────────────
  newEval = lib.evalModules {
    specialArgs.inputs = oldEval._module.specialArgs.inputs;
    modules = [
      ../modules/options/flake.nix
      ../modules/options/quad.nix
      ../modules/lib/mk-cluster-host.nix
      ../modules/lib/machine-registry.nix    # new registry module
      ../modules/outputs/nixos-configurations.nix
    ];
  };

  # ── Compare: same host names exist ────────────────────────────────────
  oldHosts = builtins.attrNames oldEval.config.flake.nixosConfigurations;
  newHosts = builtins.attrNames newEval.config.flake.nixosConfigurations;

  hostNamesMatch = (lib.sort (a: b: a < b) oldHosts) == (lib.sort (a: b: a < b) newHosts);

  # ── Compare: same role assignments ────────────────────────────────────
  # For a full store-path comparison, we'd need to actually build the
  # toplevel derivations (requires x86_64-linux platform).
  # Here we compare the module structure — same imports, same config.
  
  # As a lightweight check: verify each host has the same _quad metadata
  quadMetadataMatches = lib.all (name: let
    old = oldEval.config.quad.hosts.${name}._quad or {};
    new = newEval.config.quad.hosts.${name}._quad or {};
  in
    old.sshHost == new.sshHost &&
    old.remoteBuild == new.remoteBuild
  ) oldHosts;

in
  assert hostNamesMatch;
  assert quadMetadataMatches;
  true
```

### 5.4 Full Store-Path Golden Test (for CI/local validation)

```bash
#!/usr/bin/env bash
# tests/machines-golden-test.sh
#
# Compares old and new system toplevel store paths.
# Run on an x86_64-linux machine or with remote building enabled.
set -euo pipefail

OLD_PATH=$(nix build .#nixosConfigurations.backbone-01.config.system.build.toplevel --no-link --print-out-paths)
echo "Old backbone-01: $OLD_PATH"

# After switching flake.nix to use registry:
NEW_PATH=$(nix build .#nixosConfigurations.backbone-01.config.system.build.toplevel --no-link --print-out-paths)
echo "New backbone-01: $NEW_PATH"

if [ "$OLD_PATH" = "$NEW_PATH" ]; then
  echo "✅ PASS: Store paths identical"
else
  echo "❌ FAIL: Store paths differ"
  echo "  Old: $OLD_PATH"
  echo "  New: $NEW_PATH"
  exit 1
fi
```

### 5.5 Recommended Test to Add to `checks.nix`

```nix
# Add to nixModuleTests in checks.nix:
machines-registry = "tests/nix/machines/registry.test.nix";
machines-golden = "tests/nix/machines/golden-test.nix";
```

## 6. Migration Path

### Phase 1: Create Registry Alongside Existing System

**Goal:** Zero-impact — both systems exist, only old one is active.

1. Create `machines/default.nix` with the registry schema
2. Create `machines/hardware/` as a symlink to `modules/hardware/`
3. Create `machines/roles/` as a symlink to `modules/roles/`
4. Create `modules/lib/machine-registry.nix` (the consumer)
5. Create `tests/nix/machines/golden-test.nix`
6. Run the golden test to verify equivalence
7. **Do NOT modify** `modules/imports.nix` or `flake.nix`

**Validation:** Golden test passes. Flake still builds from old system.

### Phase 2: Switch flake.nix to Use Registry

**Goal:** Registry becomes the source of truth. Old host files become dead code.

1. Add `./lib/machine-registry.nix` to `modules/imports.nix`
2. Remove `filesIn ./hosts` from `modules/imports.nix`
3. Run `nix flake check` — all checks pass
4. Run `nix build .#nixosConfigurations.backbone-01.config.system.build.toplevel` — same store path
5. Run a test deploy with `deploy-rs -- --dry-activate`

**Validation:** Store-path comparison shows identical results. All flake checks pass.

### Phase 3: Cleanup

**Goal:** Remove old system files.

1. Delete `modules/hosts/backbone-01.nix` and `modules/hosts/frontline-01.nix`
2. Move `modules/hardware/` → `machines/hardware/` (replace symlinks)
3. Move `modules/roles/` → `machines/roles/` (replace symlinks)
4. Rename `modules/roles/frontline.nix` → `machines/roles/worker.nix`
5. Remove `filesIn ./hosts` auto-discovery from `modules/imports.nix` (done in Phase 2)
6. Delete the golden test (no longer needed)
7. Update AGENTS.md to reflect new structure

**Validation:** `nix flake check` passes. `deploy-rs` works.

## 8. Open Questions

### Q1: Role naming — "frontline" vs "worker"?

The current role is called `frontline` (in `modules/roles/frontline.nix`), but its K8s role is `["node"]` (worker). Should we rename to `worker` for clarity? The machine name stays `frontline-01` — only the role key changes.

**Recommendation:** Rename to `worker` — it's more descriptive of what the role actually does.

### Q2: Should `secrets` path be derivable?

Currently, SOPS secrets are referenced as `../../secrets/${config.networking.hostName}.yaml` in role modules. This is already dynamic. We don't need a `secrets` field in the machine registry — it's handled inside the role modules. But if a machine needs a non-standard secrets path, we'd need an escape hatch.

**Recommendation:** Don't add a `secrets` field now. The existing convention (`secrets/${hostName}.yaml`) is sufficient. Add later if needed.

### Q3: Machine-level services (like frontline's cloudflared)?

`frontline-01` has a machine-specific cloudflared tunnel in its `extraModules`. This is fine in the registry — it stays as an `extraModules` entry. But should we consider a `services` key at the machine level for common patterns?

**Recommendation:** Keep using `extraModules` for now. The DSL is meant to reduce boilerplate for the common case (adding a machine to an existing role). Machine-specific overrides are by definition not common.

### Q4: Should we validate role references?

If someone writes `role = "nonexistent"`, Nix will error with an attribute lookup failure. This is clear enough. We could add a more explicit assertion, but it's not necessary for the initial design.

**Recommendation:** Rely on Nix's natural error for missing attr keys. Can add explicit validation later.

### Q5: Keep `mkClusterHost` or inline it?

The `mkClusterHost` function in `modules/lib/mk-cluster-host.nix` is 35 lines of NixOS module assembly. It could be inlined into `machine-registry.nix`, but keeping it separate has value:
- It's independently testable (existing test at `tests/nix/quad/mk-cluster-host.test.nix`)
- It encapsulates the module system complexity
- The registry just maps data → function calls

**Recommendation:** Keep `mkClusterHost` as-is. It's a clean abstraction boundary.

## 9. Summary

| Aspect | Design |
|--------|--------|
| **Registry format** | Single `machines/default.nix` with `machines` and `roles` attrsets |
| **Role reference** | String key (`role = "backbone"`) into `roles` block |
| **Hardware** | Path reference (`hardware = ./hardware/backbone-01.nix`) |
| **Taints** | List of attrsets, passed through to `mkClusterHost` |
| **Extra modules** | List of NixOS modules for machine-specific overrides |
| **Golden test** | Compare store paths of old vs new nixosConfigurations |
| **Migration** | 3 phases: create alongside → switch flake → cleanup |
| **K8s resources** | NOT in scope — stays in `modules/outputs/bootstrap.nix` |
| **Breaking changes** | Zero — same `quad.hosts` interface, same store paths |
